Chapter One · Planning Phase

The Risk Register: Naming the Danger Before It Arrives

Before a single test is run, the team walks the process end-to-end, documents every in-scope risk, and scores it across five dimensions — FROST.

F
Financial
Estimated $ impact: <$5M (1) to >$100M (5)
R
Regulatory
Risk of fines, lawsuits, legal exposure
O
Operational
Process complexity, maturity, change
S
Strategic
Press, customer trust, reputational fallout
T
Technology
Reliance on systems; cyber & access risk
F + R + O + S + T = Inherent Risk Score × Mitigating Control Factor = Residual Risk Score
STOP RULEEvery control gets a walkthrough and a "test of one." If the design fails right there, testing stops — it's logged immediately as a missing control, not buried until fieldwork ends.

Chapter Two · Fieldwork

The Document Request List: Asking For the Truth, Precisely

Source Documents, Not Screenshots

Evidence must allow a third party to re-perform the test independently. Every source document is linked in the workpaper and stored in Google Drive — never attached directly in Optro.

Re-performable

Redaction Before Retention

All sensitive data is redacted before anything is saved. A document request is a request for truth, not for exposure — respect for the KPO's data is part of the ask.

Privacy first

Minimum Evidence Standard

For every control tested: one passing sample retained as evidence, plus documentation of every single exception found — no averaging away the outliers.

One pass + all exceptions

Scoped, Not Scattershot

Requests map directly to the Risk Register — every document asked for exists to test a specific, named risk. Nothing is requested "just in case."

No scope creep
"A sample built on a broken population is just a guess wearing a lab coat. Before anyone can trust a number, someone has to verify it's real."
Chapter Three · Fieldwork

Data Verification: Trust, But Verify — Every Time

Completeness

Reconcile the count and dollar value of the population back to the system of record. If the population is missing rows, every conclusion built on top of it is already wrong.

Accuracy

Match the data against the true source — not a report about the source. Data pulled by the entity itself (IPE) always requires this validation before it can be relied on.

C&A validation is a hard gate — no sampling begins until completeness and accuracy are both confirmed.
